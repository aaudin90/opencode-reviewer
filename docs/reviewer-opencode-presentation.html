Схема пайплайна opencode-reviewer

Что происходит от запуска CLI до публикации комментариев в VCS.

Поток выполнения

1) CLI и конфиг
`cmd/reviewer/main.go`, `internal/config/*`
Читает flags/env/TOML/.opencodereview, проверяет обязательные параметры.
↓
2) Git-данные
`internal/git/*`
Fetch ветки, определение base/head, сбор сырых diff и commit-метаданных.
↓
3) Обработка diff
`internal/diff/*`
Парсинг патча, фильтрация бинарных/нерелевантных изменений, подготовка контекста файлов.
↓
4) Оркестрация pipeline
`internal/pipeline/*`
Запуск N reviewer-сессий параллельно + запуск finalizer-сессии для дедупликации.
↓
5) Публикация результата
`internal/vcs/*`, `internal/vcs/gitlab/*`
Нормализация line-refs, формат Markdown и отправка комментариев в GitLab MR.

Какая часть за что отвечает

  • internal/config — единая модель конфигурации и приоритеты источников (CLI > ENV > TOML).
  • internal/configdir — загрузка структуры `.opencodereview` (provider, prompts, tools).
  • internal/agentconfig, providerconfig, subagentconfig — резолв prompt/provider/sub-agent конфигов.
  • internal/workspace — временная рабочая директория для артефактов сессий.
  • internal/runner — управление жизненным циклом OpenCode (`serve`/`run`).
  • internal/pipeline — бизнес-логика стадий review и финализации.
  • internal/vcs — абстракция публикации замечаний + форматирование.
  • internal/vcs/gitlab — конкретный адаптер GitLab API.
Вход: branch + config Выход: consolidated review Режим: parallel reviewers + finalizer

Схема работы с OpenCode

Как reviewer формирует контекст, запускает LLM-сессии и получает итоговый Markdown-ревью.

Интеграционный цикл OpenCode

A) Подготовка контекста
Diff и метаданные кладутся во временный workspace (`.opencode-review/diff.md`).
→
B) Инициализация OpenCode
`internal/runner` проверяет endpoint/порт, поднимает subprocess при необходимости.
→
C) Reviewer-этап
Для каждого файла из `reviewer/messages/*.md` стартует отдельная сессия OpenCode (параллельно).
→
D) Finalizer-этап
Одна сессия собирает вывод reviewer-ов, удаляет дубли, выстраивает финальный список находок.
→
E) Публикация и трассировка
Результат маппится на строки diff и отправляется в VCS-паблишер.

Контракты и ответственность слоёв

  • OpenCode provider.json определяет модель/провайдера LLM и auth-настройки.
  • Prompts разделены на роли: reviewer (поиск проблем) и finalizer (сведение/приоритизация).
  • Tools в `.opencodereview` позволяют переопределять built-in инструменты отправки review.
  • stage_timeout / max_steps ограничивают время и глубину каждой сессии.
  • Fast-review режим позволяет воспроизвести финальный результат из JSON-дампа без повторного запуска LLM.
  • Fallback-логика конфигов: приоритет config-dir, затем TOML/env (legacy-пути только как fallback).

Практическая ментальная модель

  • Reviewer = генерация кандидатов замечаний.
  • Finalizer = редактор качества и дедупликации.
  • VCS publisher = доставка в инженерный workflow команды.
